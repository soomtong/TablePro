# Import & Export

Export data in five formats (CSV, JSON, SQL, MQL, XLSX), import SQL files with gzip support, and paste tabular data from your clipboard directly into the grid.

## Export Data

### Export Options

Export data from query results or tables:

1. Run a query or open a table
2. Click the **Export** button in the toolbar (or press `Cmd+Shift+E`)
3. Choose your export format
4. Select tables to export using the tree view
5. Configure format-specific options
6. Click **Export**

<Note>
**MongoDB**: SQL export is not available for MongoDB connections. Use CSV, JSON, MQL, or XLSX instead. MQL generates `db.collection.insertMany([...])` scripts that run directly in `mongosh`.
</Note>

<Note>
**Redis**: SQL and MQL exports are not available for Redis connections. Use CSV, JSON, or XLSX to export key-value data.
</Note>

### Exporting Query Results

To export the results of a specific query to a file:

1. Write and execute your query in the SQL editor
2. Right-click on the results grid and select **Export Results...**, or go to **File** > **Export Results...**
3. Choose your export format (CSV, JSON, SQL, XLSX, or MQL)
4. Configure format-specific options and filename
5. Click **Export**

The export dialog shows the row count and uses a simplified layout without the table selection tree, since only the in-memory query results are exported.

<Tip>
For large exports, use pagination or LIMIT in your query to control the output size.
</Tip>

### Exporting Entire Tables

To export all data from a table:

1. Click on the table in the sidebar
2. Click **Export** without applying any filters
3. Or run `SELECT * FROM table_name` and export

<Tip>
Export multiple tables at once by selecting them in the export dialog's tree view. When exporting as SQL, each table can have individual options for structure (DDL), DROP statements, and data.
</Tip>

### Exporting Structure Only

To export just the table structure (DDL):

1. Open the table's **Structure** tab
2. Go to the **DDL** sub-tab
3. Copy the CREATE TABLE statement

Or use database-specific commands:

```sql
-- MySQL
SHOW CREATE TABLE users;

-- PostgreSQL
pg_dump -s -t users database_name
```

### Export Progress

When exporting large datasets, a progress indicator shows:

- **Rows processed** out of total rows
- **Current table** being exported (for multi-table exports)

## Clipboard Paste (CSV/TSV)

Paste tabular data from your clipboard directly into the data grid. TablePro auto-detects whether the content is CSV (comma-separated) or TSV (tab-separated).

**How it works:**

1. Copy data from a spreadsheet, text editor, or other source
2. Select a row in the data grid
3. Press `Cmd+V` to paste

The CSV parser is RFC 4180-compliant and handles:

- Quoted fields with commas inside (e.g., `"New York, NY"`)
- Escaped double quotes (e.g., `"He said ""hello"""`)
- Multi-line values within quoted fields
- Automatic header row detection (skips if column names match)

<Tip>
Format is auto-detected: if the data contains more tabs than commas, it parses as TSV; otherwise CSV. Data from Excel or Google Sheets is typically tab-separated.
</Tip>

## Import Data

TablePro imports **SQL files** (`.sql` and `.sql.gz`). Statements execute directly against your database, making it good for restoring backups, running migrations, or seeding data.

<Note>
**MongoDB**: SQL import is not available for MongoDB connections. Use `mongoimport` or the MQL shell for importing data.
</Note>

### Import Workflow

<Steps>
  <Step title="Open Import Dialog">
    Click **File** > **Import** (or press `Cmd+Shift+I`). You can also drag and drop a `.sql` or `.sql.gz` file onto the application.
  </Step>
  <Step title="Select SQL File">
    Choose a `.sql` or `.sql.gz` file from your filesystem.
  </Step>
  <Step title="Configure Options">
    Set encoding, transaction wrapping, and foreign key check options.
  </Step>
  <Step title="Preview SQL">
    Review the SQL code preview, statement count, and file size before importing.
  </Step>
  <Step title="Import">
    Click **Import** to execute the statements and monitor progress.
  </Step>
</Steps>

### Import Options

| Option | Description | Default |
|--------|-------------|---------|
| Encoding | File encoding: UTF-8, UTF-16, Latin1, or ASCII | UTF-8 |
| Wrap in transaction | Execute all statements within a single transaction | Yes |
| Disable foreign key checks | Temporarily disable FK constraints during import | Yes |

<Tip>
Wrapping in a transaction ensures that if any statement fails, all changes roll back, leaving your database in a consistent state.
</Tip>

<Tip>
Import options (wrap in transaction, disable foreign key checks) persist across app launches. Change them once and they stick.
</Tip>

### SQL Preview

Before running the import, the dialog shows:

- **SQL code preview** of the file contents
- **Statement count** -- total number of SQL statements detected
- **File size** of the selected file

This lets you verify the file contains the expected SQL before executing.

### Error Handling

Import stops on the first error. If the import is wrapped in a transaction, all changes roll back automatically.

When an error occurs, TablePro displays:

- **Failed statement** -- the SQL statement that caused the error
- **Line number** -- where the statement appears in the file
- **Error message** -- the error returned by the database

<Warning>
There is no "skip errors" mode. If a statement fails, the import halts. Use "Wrap in transaction" to ensure partial imports roll back cleanly.
</Warning>

## Compression

### Compressed Export

Gzip compression is available for **SQL export only**:

- Enable "Compress with gzip" in the SQL export settings
- Output is saved with a `.sql.gz` extension
- Significantly reduces file size for large SQL dumps

<Note>
Gzip compression is not available for CSV, JSON, or XLSX exports.
</Note>

### Compressed Import

- TablePro auto-detects `.sql.gz` files during import
- Files are decompressed on-the-fly -- no manual decompression needed
- Works identically to importing a regular `.sql` file

## Best Practices

### Exporting

1. **Filter first**: Export only what you need
   ```sql
   SELECT * FROM orders WHERE status = 'completed' AND year(created_at) = 2024;
   ```

2. **Consider file size**: Use LIMIT for very large tables
   ```sql
   SELECT * FROM logs ORDER BY created_at DESC LIMIT 100000;
   ```

3. **Choose appropriate format**:
   - CSV for data analysis tools
   - JSON for APIs/web
   - SQL for database migration
   - MQL for MongoDB backup/restore via `mongosh`
   - XLSX for sharing with non-technical users in Excel

4. **Use gzip for large SQL exports**: Compression reduces file size significantly

### Importing

1. **Backup first**: Always backup your database before running SQL imports

2. **Test with a small file first**: Verify your import settings on a subset before running a large import

3. **Use transactions**: Keep "Wrap in transaction" enabled so failed imports roll back automatically

4. **Disable foreign key checks**: When importing dumps that reference tables in a specific order, keep this enabled to avoid constraint errors

5. **Check encoding**: Make sure the encoding setting matches your SQL file's actual encoding to avoid character corruption

## Troubleshooting

### Export Issues

**"File too large"**
- Use LIMIT in your query
- Export in batches
- Enable gzip compression for SQL exports
- Choose a more compact format (CSV vs JSON)

**"Character encoding issues"**
- Verify database encoding
- Export with UTF-8 encoding
- Check for special characters

### Import Issues

**"Syntax error in SQL"**
- Verify the SQL file was exported from a compatible database engine
- Check that the file encoding matches the setting in the import dialog
- Open the file in a text editor to inspect the problematic line

**"Duplicate key error"**
- The data contains duplicates for unique or primary key columns
- Include DROP TABLE statements in your SQL export to start fresh
- Or manually drop the target table before importing

**"Foreign key constraint"**
- Enable "Disable foreign key checks" in the import options
- Or ensure parent tables are created before child tables in your SQL file

**"Transaction rolled back"**
- A statement in the file caused an error, rolling back the entire import
- Check the error message and line number to identify the problematic statement
- Fix the SQL file and re-import
